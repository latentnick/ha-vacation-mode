## Project Overview

This project fetches historical light usage data from Home Assistant via InfluxDB, trains an LSTM model to learn daily/weekly patterns, generates a realistic vacation light schedule, and executes it via the Home Assistant REST API. The goal is to make a home look occupied while the owner is away.

## Architecture

### Data flow
1. `fetch_ha_data.py` — pulls state history from InfluxDB, writes `out/data.csv` and `out/entity_map.json`
2. `lights.ipynb` — trains LSTM, generates schedule, writes `out/schedule_events.json`
3. `vacation_daemon.py` — reads `out/schedule_events.json`, calls HA REST API at each event time
4. `run.sh` — orchestrates steps 1 and 2 with vacation dates as arguments

### Key files
| File | Purpose |
|------|---------|
| `config.json` | List of HA entity IDs to control (gitignored, copy from `config.json.example`) |
| `.env` | Credentials for HA and InfluxDB (gitignored, copy from `.env.example`) |
| `configure.py` | Interactive script to query HA and generate `config.json` |
| `out/entity_map.json` | Maps InfluxDB entity names → HA entity IDs (generated by fetch) |
| `out/schedule_events.json` | On/off events with timestamps (generated by notebook) |
| `out/lights_executed.ipynb` | Executed notebook with all outputs for review |

## Development Environment

### Package manager
Use `uv` for everything. Run scripts with `uv run <script>`, add packages with `uv add <package>`. Python version is pinned to 3.12 in `.python-version` (TensorFlow does not support 3.14+).

### Jupyter kernel
The notebook uses a kernel named `lights` registered from the uv venv:
```bash
uv run python -m ipykernel install --user --name lights --display-name "lights"
```

### Running the notebook headlessly
```bash
uv run papermill lights.ipynb out/lights_executed.ipynb --kernel lights \
  -p VACATION_START '2026-04-01' \
  -p VACATION_END '2026-04-08'
```

## Home Assistant Integration

### InfluxDB (v1.8.x)
- Add-on: `hassio-addons/addon-influxdb`
- Uses InfluxQL (not Flux)
- Credentials are set in the addon's Chronograf UI, not HA login credentials
- Light states stored in the `state` measurement with `entity_id` field and `state` field (`"on"`/`"off"`)

### REST API
- Service calls: `POST /api/services/switch/turn_on` and `turn_off`
- Auth: `Authorization: Bearer <token>` header
- Token from: HA Profile → Security → Long-Lived Access Tokens

### Entity naming
- InfluxDB stores entities without domain prefix (e.g. `office_main_lights`)
- HA REST API uses full entity IDs with domain (e.g. `switch.office_main_lights`)
- `out/entity_map.json` holds the mapping between the two

## Data Structure

### out/data.csv
- `time`: ISO 8601 timestamp, UTC from InfluxDB, converted to US/Pacific in the notebook
- `entity_id`: short name without domain (e.g. `office_main_lights`)
- `state.value`: `"1"` = on, `"0"` = off

### out/schedule_events.json
List of `{time, entity_id, action}` objects sorted by time. `entity_id` is the full HA entity ID (e.g. `switch.office_main_lights`). `action` is `"turn_on"` or `"turn_off"`.

## Model

- Architecture: 2-layer LSTM → Dense → sigmoid output per light
- Input: 12-step (3-hour) sequence of light states + temporal features (hour sin/cos, day sin/cos, is_weekend, total_lights_on)
- Output: probability of each light being on at the next 15-minute interval
- Timestamps must be in Pacific time before feature engineering — InfluxDB returns UTC
- Temperature scaling in `generate_vacation_schedule()` works via logit space: values <1 are more deterministic, >1 are more random
- All output saved to `out/` directory (gitignored)
