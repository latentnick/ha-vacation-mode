## Project Overview

This project fetches historical light usage data from Home Assistant via InfluxDB, generates a realistic vacation light schedule via empirical-day resampling, and executes it via the Home Assistant REST API. The goal is to make a home look occupied while the owner is away.

## Architecture

### Data flow
1. `fetch_ha_data.py` — pulls state history from InfluxDB, writes `out/data.csv` and `out/entity_map.json`
2. `generate_resample.py` — for each block of each vacation day, samples a real historical day (day-of-week match, weighted by seasonal proximity) and replays that donor's switch events within the block with small jitter. Writes `out/schedule_events.json`.
3. `vacation_daemon.py` — reads `out/schedule_events.json`, calls HA REST API at each event time
4. `generate_model.sh` — orchestrates steps 1 and 2 with vacation dates as arguments

### Key files
| File | Purpose |
|------|---------|
| `config.json` | List of HA entity IDs to control (gitignored, copy from `config.json.example`) |
| `.env` | Credentials for HA and InfluxDB (gitignored, copy from `.env.example`) |
| `configure.py` | Interactive script to query HA and generate `config.json` |
| `out/entity_map.json` | Maps InfluxDB entity names → HA entity IDs (generated by fetch) |
| `out/schedule_events.json` | On/off events with timestamps (generated by `generate_resample.py`) |

## Development Environment

### Package manager
Use `uv` for everything. Run scripts with `uv run <script>`, add packages with `uv add <package>`. Python version is pinned to 3.12 in `.python-version`.

## Home Assistant Integration

### InfluxDB (v1.8.x)
- Add-on: `hassio-addons/addon-influxdb`
- Uses InfluxQL (not Flux)
- Credentials are set in the addon's Chronograf UI, not HA login credentials
- Light states stored in the `state` measurement with `entity_id` field and `state` field (`"on"`/`"off"`)

### REST API
- Service calls: `POST /api/services/switch/turn_on` and `turn_off`
- Auth: `Authorization: Bearer <token>` header
- Token from: HA Profile → Security → Long-Lived Access Tokens

### Entity naming
- InfluxDB stores entities without domain prefix (e.g. `office_main_lights`)
- HA REST API uses full entity IDs with domain (e.g. `switch.office_main_lights`)
- `out/entity_map.json` holds the mapping between the two

## Data Structure

### out/data.csv
- `time`: ISO 8601 timestamp, UTC from InfluxDB, converted to US/Pacific in the generator
- `entity_id`: short name without domain (e.g. `office_main_lights`)
- `state.value`: `"1"` = on, `"0"` = off

### out/schedule_events.json
List of `{time, entity_id, action}` objects sorted by time. `entity_id` is the full HA entity ID (e.g. `switch.office_main_lights`). `action` is `"turn_on"` or `"turn_off"`.

## Generator

`generate_resample.py` draws an independent donor day per time block within each vacation day (default 2 blocks = morning/evening split). For each block it aligns simulated state to the donor's state at the block boundary, then replays the donor's switch events within that block with ±10 minute jitter. A final pass after global sort drops events that don't change state (jitter can reorder events across midnight).

Empirically this beats whole-day replay (`--blocks 1`) and finer splits on hourly on-rate MAD, KL divergence vs. history, and switch-count error.
